  <behavior>
    ConnectionManager:
    - `ConnectionManager(bus: EventBus)` constructs with an empty set of client queues.
    - `await manager.connect(websocket)` calls `await websocket.accept()`, allocates a new `asyncio.Queue`, adds it to `self._clients`, and returns the queue.
    - `manager.disconnect(queue)` removes the queue from `self._clients` (no error if already absent — use `discard`).
    - `await manager.start_background_fan_out()` subscribes to all 7 EventBus topics (`TOPIC_BARS`, `TOPIC_SIGNALS`, `TOPIC_RISK_DECISIONS`, `TOPIC_FILLS`, `TOPIC_POSITIONS`, `TOPIC_EQUITY`, `TOPIC_DEGRADED_STATE`) concurrently via `asyncio.gather`. For each event received on any topic, it JSON-serializes the D-05 envelope `{"type": event.topic, "payload": event.model_dump(mode="json")}` and puts the resulting string onto every connected client queue.
    - When the fan-out task is cancelled (lifespan shutdown), it raises `asyncio.CancelledError` and exits cleanly.

    WS /stream route:
    - On client connect, the route calls `q = await app.state.manager.connect(websocket)`, then enters `while True: msg = await q.get(); await websocket.send_text(msg)`.
    - On `WebSocketDisconnect` exception, the route calls `app.state.manager.disconnect(q)` and exits cleanly (no log spam — disconnects are normal).

    Envelope + topic coverage (D-04 / D-05):
    - For every event published on each of the 7 topics, the WS client receives a JSON message with exactly two top-level keys: `type` (str matching the topic name) and `payload` (an object — the serialized event Pydantic model).
    - `BarReceived` event published on TOPIC_BARS → client receives `{"type":"bars","payload":{"topic":"bars","emitted_at":"...","bar":{...}}}` (the topic shows up twice — once as the envelope `type`, once inside the payload because `Event.topic` is a field; this duplication is acceptable for v1 and documented).
    - `DegradedStateEvent` published on TOPIC_DEGRADED_STATE → client receives `{"type":"degraded_state","payload":{"topic":"degraded_state","emitted_at":"...","source":"tradingview_mcp","reason":"CDP disconnect"}}`.

    Single-client + multi-client invariants:
    - A single connected client receives a published event within 200ms.
    - Two concurrent clients BOTH receive a single published event (per-client queue isolation — Pattern from 03-RESEARCH.md).
    - Disconnecting one client does not affect the other (queue is removed from the set; future publishes only enqueue on remaining clients).
  </behavior>
