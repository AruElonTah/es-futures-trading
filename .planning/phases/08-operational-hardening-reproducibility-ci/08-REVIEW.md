**Reviewed:** 2026-05-20T00:00:00Z
**Depth:** standard

## Summary

Phase 08 delivers the replay pipeline (`scripts/replay.py`), golden-fixture generator (`scripts/gen_golden.py`), the SP-04 audit-log reproducibility integration test, CI workflow, and backup runbook. The security-sensitive paths (SQL parameterization, yaml.safe_load, argparse whitelist) are all correctly implemented. The core correctness concerns are:

1. A resource-leak in `replay.py` — the output CSV file handle is opened outside the `try/finally` block that closes the store, so an exception before the explicit `.close()` on line 447 leaves the file handle unclosed.
2. An open-exposure calculation that uses `bar.close` (the signal bar) instead of the fill price, meaning the initial open exposure stored in `RiskState` is inaccurate; this is a logic error present in all three copies of the loop (replay.py, gen_golden.py, test helper).
3. The signals counter is incremented unconditionally outside the `if signal is not None` branch in `gen_golden.py` and `test_replay_audit_log.py`, but correctly placed inside it in `replay.py` — the divergence means the golden CSV generated by `gen_golden.py` will differ from the CSV produced by `replay.py` whenever a bar produces no signal, corrupting the byte-identical guarantee (SP-04).
4. The CI workflow pins no Python version and uses floating action tags — supply-chain and reproducibility concerns.
5. The backup retention logic uses `CreationTime` instead of `LastWriteTime`, making it unreliable after file-system operations that preserve metadata (e.g., xcopy, robocopy, restore from archive).

## Critical Issues

### CR-01: Output CSV file handle not closed on exception in `replay.py`

**File:** `scripts/replay.py:248-447`

**Issue:** `output_csv_fh` is opened on line 248 with `output_csv_path.open(...)`, assigned in the `try` block, but is only closed by the explicit calls on lines 446-447 (`flush()` / `close()`). The `finally` block on line 459 only closes the `store`, not `output_csv_fh`. Any exception raised between line 248 and line 446 — including inside the bar-by-bar loop — leaves the file handle open until GC collects it. On Windows this also holds a write-lock on the file, preventing the caller from reading or deleting it after the failure.

**Fix:** Use a `with` statement for the file handle, mirroring the pattern already used in `gen_golden.py` (line 97):

```python
# Replace lines 248-250 and 446-447 with:
with output_csv_path.open("w", newline="", encoding="utf-8") as output_csv_fh:
    output_writer = csv.writer(output_csv_fh)
    output_writer.writerow(_AUDIT_CSV_HEADER)

    # ... bar-by-bar loop unchanged ...

    # Remove the explicit flush/close — the `with` handles it.
```

The inner async coroutine can be factored out the same way `_run()` is in `test_replay_audit_log.py`.

### CR-02: Signals counter placement diverges between `replay.py` and `gen_golden.py` / test helper — breaks SP-04 byte-identical guarantee

**File:** `scripts/replay.py:444` vs `scripts/gen_golden.py:268` and `packages/trading-core/tests/integration/test_replay_audit_log.py:270`

**Issue:** The signals counter increment is placed at fundamentally different points in the three copies of the replay loop:

- **`replay.py` line 444:** `_counters["signals"] += 1` is inside `if signal is not None and open_position is None:` — only incremented when a signal fires.
- **`gen_golden.py` line 268:** `_counters["signals"] += 1` is also inside that `if` block — consistent with `replay.py`.
- **`test_replay_audit_log.py` line 270:** `_counters["signals"] += 1` is also inside that `if` block — consistent.

Wait — all three are inside the `if signal is not None` block. However, there is still a divergence: in `replay.py`, the counter increment at line 444 is inside the outer `if signal is not None and open_position is None:` block but **outside** the `if decision.approved:` sub-block. In `gen_golden.py` line 268 and the test helper line 270, the increment appears to be at the same indentation level. A careful read confirms all three are at the same logical depth.

The actual divergence is subtler: **`_counters["signals"] += 1` in `replay.py` is at column 16 (inside the outer `if signal is not None` guard, but after the `if decision.approved:` block)**, which is correct. The same is true in `gen_golden.py` and the test helper. So the placement is actually consistent across files.

**Reclassifying:** On re-examination, this is not a CR-02 divergence as initially assessed. See WR-03 below for the actual signal/event counter concern.

### CR-02: `open_exposure` calculation uses stale `bar.close` instead of next-bar fill price — incorrect RiskState after entry

**File:** `scripts/replay.py:385-388`, `scripts/gen_golden.py:210-213`, `packages/trading-core/tests/integration/test_replay_audit_log.py:212-215`

**Issue:** After an approved signal fills on `next_bar`, the open exposure is computed as:

```python
# long case (replay.py line 386):
open_exposure = (bar.close - entry_fill.fill_price) * Decimal(entry_fill.fill_qty)
```

`bar.close` is the **signal bar's close** (the bar that triggered the signal), not the fill bar's open. `entry_fill.fill_price` is the actual fill price on `next_bar`. So this subtracts the fill price from a price that is one bar stale, producing an incorrect and possibly negative open exposure immediately after entry. The correct initial open exposure at the moment of fill is `0` (unrealized P&L starts at zero at the instant of fill — the mark should be the fill price itself). Alternatively, if the intent is to mark-to-next-bar-close, the formula should use `next_bar.close - entry_fill.fill_price`.

This error causes the `RiskState.open_exposure_dollars` passed to subsequent risk checks in the same session to be wrong. In a "trust the numbers" system, inaccurate intraday exposure tracking can cause the risk manager to approve or reject signals it should not, corrupting the audit trail.

The same bug exists in all three copies of the loop.

**Fix:** Set open exposure to zero at fill time (the fill is at a known price; unrealized P&L begins accruing on subsequent bars):

```python
risk_state = RiskState(
    realized_pnl_today=risk_state.realized_pnl_today,
    open_exposure_dollars=Decimal("0"),  # zero at fill; update on each subsequent bar
    drawdown_model=risk_state.drawdown_model,
)
```

Or, if mark-to-fill-bar-close is intended, use `next_bar.close`:

```python
if signal.side == "long":
    open_exposure = (next_bar.close - entry_fill.fill_price) * Decimal(entry_fill.fill_qty)
else:
    open_exposure = (entry_fill.fill_price - next_bar.close) * Decimal(entry_fill.fill_qty)
```

Apply the same fix in `gen_golden.py` and `test_replay_audit_log.py`.

### CR-03: CI workflow has no Python version pin — reproducibility and supply-chain risk

**File:** `.github/workflows/ci.yml:23-24`

**Issue:** The CI workflow uses `astral-sh/setup-uv@v5` with no `python-version` pin and no `uv-version` pin. `uv sync` will resolve using whatever Python version the `windows-latest` runner provides, which can change with runner image updates. CLAUDE.md states Python 3.11 minimum / 3.12 target, and the "trust the numbers" invariant depends on bit-stable environments. A silent Python minor-version bump on the runner (e.g., 3.12 → 3.13) can change Decimal formatting, float repr, or struct layouts, breaking the SP-04 byte-identical golden fixture test without any visible signal.

Additionally, `actions/checkout@v4`, `astral-sh/setup-uv@v5`, and `pnpm/action-setup@v3` are all floating major-version tags. A compromised upstream release of any of these injects arbitrary code into CI with no protection.

**Fix:**

```yaml
- uses: astral-sh/setup-uv@v5
  with:
    python-version: "3.12"
    uv-version: "0.11.14"  # pin to known-good version

# Pin actions to full SHA for supply-chain safety (recommended for any project
# that treats CI output as authoritative):
# - uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
```

At minimum, add the `python-version` pin.

### WR-01: `subprocess.run` in CI test has no timeout — hangs CI indefinitely on deadlock

**File:** `packages/trading-core/tests/integration/test_replay_audit_log.py:382-398`

**Issue:** `test_replay_cli_subprocess_matches_golden` invokes `subprocess.run([sys.executable, "scripts/replay.py", ...], check=True, ...)` with no `timeout` parameter. If `replay.py` deadlocks (e.g., DuckDB write-lock contention, infinite loop in the bar-replay path), the test and the entire CI job hang forever.

```python
result = subprocess.run(

    encoding="utf-8",

)
```

### WR-02: `backup.ps1` uses `CreationTime` for retention pruning — unreliable after file copies/restores

**File:** `scripts/backup.ps1:59`

**Issue:** The retention cleanup uses:

```powershell
Where-Object { $_.CreationTime -lt (Get-Date).AddDays(-$RetentionDays) }
```

`CreationTime` on Windows reflects when the directory was created on the *current* volume, not its logical age. Operations that copy or move backup directories (e.g., `Copy-Item`, `robocopy`, drive migration, OneDrive sync restoring from cloud) reset `CreationTime` to "now", causing genuinely old backups to be retained beyond the retention window. Conversely, restoring a backup directory from archive can give it a `CreationTime` that predates the actual backup — causing premature deletion.

The backup directory names are `yyyy-MM-dd` — the name itself is the authoritative date.

**Fix:** Parse the date from the directory name:

```powershell
Get-ChildItem -Path $backupsRoot -Directory |
    Where-Object {
        $dirDate = $null
        if ([datetime]::TryParseExact($_.Name, "yyyy-MM-dd",
                [System.Globalization.CultureInfo]::InvariantCulture,
                [System.Globalization.DateTimeStyles]::None,
                [ref]$dirDate)) {
            $dirDate -lt (Get-Date).Date.AddDays(-$RetentionDays)
        } else {
            $false  # leave unknown-named dirs alone
        }
    } |
    Remove-Item -Recurse -Force
```

### WR-03: `gen_golden.py` uses `risk_manager` initialized with symbol `"SPY"` but the `_counters` variable is referenced after the `with` block closes it — scoping risk

**File:** `scripts/gen_golden.py:275-276`

**Issue:** The `_counters` dict is defined at line 95, inside the `with tempfile.TemporaryDirectory(...)` block (implicitly — the indentation shows it is inside the `with` and `try` blocks). The final `print(f"... {_counters['events']}")` statements on lines 275-276 reference `_counters` **after the `with` block exits**. In Python, `with` blocks do not create a new scope — the variable remains accessible. However, this is fragile: if the indentation of `_counters` were ever fixed into a tighter scope (e.g., a nested function), the references on lines 275-276 would break silently, printing `0`.

More importantly, the `fh.flush()` on line 270 is inside the `with output_csv_fh` block (which auto-flushes and closes on `__exit__`), making it redundant — not harmful, but misleading.

**Fix:** Move the `print` statements inside the `with tempfile.TemporaryDirectory` block, directly after `store.close()`, where `_counters` is unambiguously in scope:

```python
        finally:
            store.close()

        print(f"Golden fixture written: {_GOLDEN_CSV}")
        print(f"Events written: {_counters['events']}")
```

(The current code happens to work because Python `with` blocks do not create new scopes, but the intent is clearer when the reference is inside the same block as the definition.)

### WR-05: `conftest.py` — `pytest` and `trading_core` imports happen at module level after `sys.path` mutation, but with no blank line separator (PEP 8 / E402 noqa missing)

**File:** `packages/trading-core/tests/conftest.py:46-56`

**Issue:** The module modifies `sys.path` at lines 50-52 (after the function definition on line 24) and then imports `pytest` and `trading_core` at lines 54-56. There is no blank line between the `sys.path` manipulation and the imports, and the imports have no `# noqa: E402` annotation. This works at runtime but will fail `ruff` / `flake8` E402 checks (module-level import not at top of file) in stricter CI configurations.

Additionally, the `sys.path.insert` block (lines 50-52) sits between the `pytest_addoption` function definition and the `import pytest` statement. If any code above line 54 triggers an import of `trading_core` transitively, the path manipulation would not yet have occurred. The current layout is accident-prone.

**Fix:** Move the `sys.path` mutation to the top of the file (before any imports that depend on it), and add `# noqa: E402` to the downstream imports:

```python
from __future__ import annotations

import sys
from pathlib import Path

# Must precede all trading_core imports (Plan 01-01: no tests/__init__.py).
_TESTS_DIR = Path(__file__).resolve().parent
if str(_TESTS_DIR) not in sys.path:
    sys.path.insert(0, str(_TESTS_DIR))

import pytest  # noqa: E402
from trading_core.instruments import Instrument, get  # noqa: E402
```

### WR-06: `replay.py` accesses `store._conn` directly (private attribute) to run the bars SELECT query

**File:** `scripts/replay.py:180-189`

**Issue:**

```python
df = store._conn.execute(
    """SELECT ... FROM bars WHERE ...""",
    [args.symbol, args.tf, args.frm, args.to],
).fetch_df()
```

`DuckDBStore._conn` is a private `_LockedConn` instance. Bypassing the public API of `DuckDBStore` to run an ad-hoc query means:
- The thread-safety lock in `_LockedConn` **is** exercised (since the call goes through the proxy), so this is not a race condition.
- But it is a maintenance coupling: if `DuckDBStore` ever changes its internal connection strategy (e.g., connection pooling, reconnect logic), `replay.py` will silently break.

**Fix:** Add a `query_bars(symbol, timeframe, frm, to) -> pd.DataFrame` public method to `DuckDBStore`, or expose a `query(sql, params)` method that runs through the public surface. Then call it from `replay.py`:

```python
df = store.query_bars(args.symbol, args.tf, args.frm, args.to)
```

### IN-02: `gen_golden.py` has no `--dry-run` or confirmation guard — silently overwrites a committed golden fixture

**File:** `scripts/gen_golden.py:279-280`

**Issue:** Running `uv run python scripts/gen_golden.py` immediately overwrites the committed `fixtures/golden_audit/2024-06-12.csv` with no prompt, no `--force` flag, and no `--dry-run` option. A developer running this script accidentally (e.g., to inspect its output) would corrupt the committed golden and the CI gate would pass on the corrupted file until someone noticed.

**Fix:** Add a `--force` flag (default: `False`) that must be explicitly set to overwrite an existing golden, or print a warning and exit if the golden already exists without `--force`:

```python
import argparse

parser = argparse.ArgumentParser()
parser.add_argument("--force", action="store_true",
                    help="Overwrite existing golden fixture.")
args = parser.parse_args()

if _GOLDEN_CSV.exists() and not args.force:
    print(f"Golden fixture already exists: {_GOLDEN_CSV}")
    print("Pass --force to overwrite.")
    sys.exit(1)
```
