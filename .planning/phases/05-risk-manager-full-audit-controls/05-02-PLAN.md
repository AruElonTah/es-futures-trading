Sizing formula (RM-01 unit test requirement):
- size_for_stop(risk_dollars=1000, stop_ticks=5, instrument=MES) == 40
  Proof: floor(1000 / (5 * 5.00)) = floor(1000 / 25) = 40
  Uses instrument.point_value (NOT tick_value — "stop_ticks" in RM-01 is "stop in index points")
- size_for_stop(risk_dollars=1000, stop_ticks=5, instrument=ES) == 4
  Proof: floor(1000 / (5 * 50.00)) = floor(1000 / 250) = 4

stop_distance in points = abs(entry - stop) / instrument.tick_size * instrument.tick_size ...

So stop_ticks_for_signal = abs(signal.entry - signal.stop) for a signal whose prices are in
index points (ES/MES). Then worst_case_loss = stop_ticks_for_signal * instrument.point_value * size.

    - size_for_stop(risk_dollars=Decimal("1000"), stop_ticks=Decimal("5"), instrument=MES) == 40
    - size_for_stop(risk_dollars=Decimal("1000"), stop_ticks=Decimal("5"), instrument=ES) == 4
    - Formula: floor(risk_dollars / (stop_ticks * instrument.point_value))
    - Returns int (uses math.floor on Decimal arithmetic)

    RM-06 — max_contracts cap:
    - size_for_stop(Decimal("100000"), Decimal("1"), MES) uncapped would be 20000,
      but min(20000, config.max_contracts=2) returns 2
    - Cap is applied AFTER size_for_stop math, not inside it

    RM-02 — all three DrawdownModel variants tracked side-by-side:
    - STATIC: hwm never decreases; floor = hwm - daily_dd_limit (set once at session start)
    - TRAILING_EOD: hwm updates at session close only (not intraday); floor = hwm - daily_dd_limit
    - TRAILING_INTRADAY: hwm updates on every fill or equity tick; floor = hwm - daily_dd_limit
    - All three HWM values written to risk_state on every check() call

    RM-03 — worst_case_loss check:
    - worst_case_loss = abs(signal.entry - signal.stop) * instrument.point_value * proposed_size
    - If (active_dd_model_floor - current_equity) < worst_case_loss: reject with reason='dd_floor_violation'
    - Active DD model determined by config.drawdown_model

    RM-04 — daily-DD circuit breaker:
    - If (realized_pnl_today + open_exposure_dollars) <= -daily_dd_limit: reject with reason='daily_dd_breaker'
    - Existing positions NOT closed (kill-switch behavior, not flatten)

    D-10 — asyncio.Event kill-switch gate:
    - When _kill_event is set, check() immediately returns RiskDecision(approved=False, reason='kill_switch_active', adjusted_size=0)
    - This check happens BEFORE any other logic (concurrency cap, DD checks, sizing)
    - set_killed(state='killed') sets the event; set_killed(state='running') or set_killed(state='paused') clears it

    RM-08 — per-strategy concurrency cap (v1 = 1 active position per strategy_id):
    - Second signal from same strategy_id while first position is open: reject with reason='concurrency_cap'
    - FullRiskManager maintains _positions: dict[str, dict] (strategy_ids -> full position metadata)
    - Concurrency check uses: if signal.strategy_id in self._positions
    - record_position_open(strategy_id, position_info: dict) and record_position_closed(strategy_id) methods to update the dict
    - position_info dict keys: symbol, strategy_id, side, qty, avg_fill, mark, stop, target, entry_ts_utc

    Signal approved path:
    - All checks pass -> approved=True, reason='approved', adjusted_size=min(sized, max_contracts)
    - write_risk_state() and write_audit_event() called BEFORE returning RiskDecision (SP-03 kill-9 guarantee)
    - write_audit_event() called with topic=TOPIC_RISK_DECISIONS, reason_code='approved' or rejection reason
    - DuckDB writes also happen on rejection paths (every check() call is audited)

    Audit write ordering (WARNING fix — SP-03):
    - write_risk_state() called FIRST (before returning)
    - write_audit_event() called SECOND (before returning)
    - THEN return RiskDecision
    - This ensures kill -9 after check() has already committed the audit row

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| signal -> check() | Signal values (entry, stop, size_hint) come from strategy; must be validated |
| check() -> DuckDB | Write path; failure must not silently pass bad data |
| _positions dict | In-memory only; lost on process restart — acceptable for v1 paper mode |
| _kill_event | asyncio.Event; fast in-process signal; bootstrapped from DuckDB on startup |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Spoofing | signal.strategy_id | accept | Single-operator local; strategy_id is set by trusted strategy code, not user input |
| T-05-02-02 | Tampering | size_for_stop arithmetic | mitigate | Pure function with Decimal-only arithmetic; unit tests lock the exact output for (1000, 5, MES/ES) |
| T-05-02-03 | Repudiation | RiskDecision not persisted | mitigate | Every check() result written to audit_log with reason_code; write happens BEFORE coroutine returns (SP-03) |
| T-05-02-04 | Denial of Service | DB write in hot path | accept | DuckDB writes at 1m bar rates are < 1ms; paper mode has no performance SLA |
| T-05-02-05 | Elevation of Privilege | concurrency_cap bypass | mitigate | _positions dict checked atomically in check(); asyncio single-threaded so no race |
| T-05-02-06 | Denial of Service | kill-switch Event not set on restart | mitigate | load_kill_state_from_db() bootstraps asyncio.Event from DuckDB on lifespan startup (D-10) |
