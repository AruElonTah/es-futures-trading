    Read the current ws-reconnect.spec.ts (currently a stub with test.fixme). Replace it with a real test.

    The test must implement D-21: simulate WS disconnect for ~5 seconds (not 30 — a shorter drop is equivalent for testing reconnect logic and much faster in CI), observe reconnect, assert no stale data.

    Test structure:
    ```
    import { test, expect } from '@playwright/test'

    test('WS reconnects after network drop and shows no permanent stale data', async ({ page }) => {
      // Step 1: Navigate to dashboard
      await page.goto('/dashboard')
      // Wait for ConnectionStatus to indicate connected state
      // The ConnectionStatus component shows a green-ish indicator when WS is connected
      // Look for the connection status text or color indicator
      await page.waitForSelector('[data-testid="connection-status"]', { timeout: 15000 }).catch(() => {
        // If no test-id, wait for the page to be loaded and WS to have connected
      })

      // Step 2: Intercept and abort WS connection via routeWebSocket
      // page.routeWebSocket intercepts all WS connections matching the pattern
      let wsRoute: any = null
      await page.routeWebSocket('**/stream', (ws) => {
        wsRoute = ws
        // Connect to the real server first so the page sees a real connection
        ws.connectToServer()
        // Then abort after a brief moment to simulate disconnect
        setTimeout(() => {
          ws.close()
        }, 500)
      })

      // Step 3: Wait for reconnect attempt (backoff delay is short — first attempt is ~1s)
      await page.waitForTimeout(4000)

      // Step 4: Release the route so subsequent reconnects work
      await page.unrouteWebSocket('**/stream')

      // Wait for reconnect to succeed
      await page.waitForTimeout(3000)

      // Step 5: Assert dashboard has no permanent stale data
      // The blotter pane should render (even if showing empty state)
      await expect(page.locator('text=BLOTTER').first()).toBeVisible()
      await expect(page.locator('text=HISTORY').first()).toBeVisible()
      await expect(page.locator('text=CONTROLS').first()).toBeVisible()

      // The API positions endpoint should still respond (no permanent failure state)
      const positionsResponse = await page.evaluate(async () => {
        const r = await fetch('http://localhost:8000/positions')
        return r.status
      })
      expect([200, 404]).toContain(positionsResponse)  // 200 normal, 404 if no positions — both ok
    })
    ```

    Add data-testid="connection-status" to the ConnectionStatus component if it doesn't already have it — read ConnectionStatus.tsx to check. If adding the attribute would cause TypeScript issues, use aria-label or a simpler locator strategy (e.g., page.locator('[aria-label="connection status"]')).

    Note: If the Next.js dev server is not running during test execution, page.routeWebSocket will not match real WS connections. The Playwright config's webServer block handles this. The test may time out in some CI environments — it is acceptable to mark it with test.slow() to get a longer timeout.

    The test should pass when uvicorn is running on port 8000 and Next.js dev server on port 3000. In CI without the servers: the webServer config in playwright.config.ts starts pnpm dev; uvicorn must be started separately or the test gracefully handles API unavailability via the expect([200, 404]) pattern.
