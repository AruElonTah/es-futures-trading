# BLOCKING CONSTRAINTS — Read Before Anything Else

> These are not suggestions. Each constraint below was discovered through failure.
> Acknowledge each one explicitly before proceeding.

- [ ] CONSTRAINT: StrictMode double-mount + chart.remove() — React StrictMode fires effects twice (create → cleanup → create). `chart.remove()` on cleanup clears lightweight-charts' inline container `width`/`height` styles. A subsequent `createChart()` call that reads `clientHeight` gets `0`. **Mitigation already applied**: Chart.tsx now uses `getBoundingClientRect()` for initial sizing AND splits into two effects — see anti-patterns table below.

- [ ] CONSTRAINT: `trades` in Effect deps caused chart recreate — When `trades` was in the same `useEffect` deps as `bars`, a late-arriving trades query fired a second chart create/destroy cycle. The second `createChart()` call read `clientHeight=0`. **Mitigation already applied**: trades are now handled in Effect 2 (overlay-only, no chart lifecycle), keyed off `chartKey` + `seriesRef`.

**Do not proceed until all boxes are checked.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| `clientHeight` after `chart.remove()` | `chart.remove()` clears inline styles on the container. A subsequent `createChart()` reading `clientHeight` gets `0`. Caused invisible charts on refresh. | blocking | Use `getBoundingClientRect()` for all initial sizing — forces synchronous layout flush, returns live dimensions regardless of inline style state. |
| Mixing chart lifecycle + trade overlays in one effect | Adding `trades` to the chart-lifecycle useEffect deps caused chart destroy/recreate when trades arrived after bars, triggering the `clientHeight=0` bug on every trade query. | blocking | Two-effect pattern: Effect 1 (deps: bars, orbHigh, orbLow) owns chart lifecycle. Effect 2 (deps: trades, chartKey) applies overlays imperatively via `seriesRef.current`. |
| `_find_repo_root` returning first pyproject.toml | Original backtests.py walked upward and returned the FIRST pyproject.toml found — could be an inner package root, not the repo root. Caused `_EQUITY_ROOT` to point to a wrong path. | advisory | Fixed in both backtests.py and optimizations.py: walk continues to find the OUTERMOST pyproject.toml. |
| DuckDB write connection held open during ProcessPoolExecutor dispatch | Workers opening read-only connections failed when the orchestrator held the write connection. | advisory | run_opt.py: `store.close()` before `ProcessPoolExecutor`, reopen after all futures complete. |

<decisions_made>

- **Two-effect Chart.tsx architecture**: Effect 1 owns chart lifecycle (deps: bars, orbHigh, orbLow). Effect 2 applies trade overlays via `seriesRef.current` (deps: trades, chartKey). `chartKey` is a state counter incremented by Effect 1 after each successful chart create — signals Effect 2 that a new series is ready, handling both orderings (trades-before-bars and trades-after-bars).

- **`getBoundingClientRect()` for initial sizing**: Forces synchronous layout flush. Returns live dimensions even after `chart.remove()` clears inline styles. Used in both Chart.tsx and EquityCurve.tsx.

- **`_find_repo_root` outermost walk**: Both backtests.py and optimizations.py now walk to the outermost directory containing pyproject.toml (repo root), not the first one. Prevents `_EQUITY_ROOT`/`_OPT_EQUITY_ROOT` pointing to a package subdirectory.

- **run_opt.py DuckDB close/reopen**: `store.close()` before `ProcessPoolExecutor.__enter__()`, `DuckDBStore(duckdb_path)` after all futures complete. Error handler also reopens if `store is None`.
</decisions_made>
