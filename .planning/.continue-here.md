# BLOCKING CONSTRAINTS — Read Before Anything Else

> These are not suggestions. Each constraint below was discovered through failure.
> Acknowledge each one explicitly before proceeding.

- [ ] CONSTRAINT: StrictMode double-mount + chart.remove() — React StrictMode fires effects twice (create → cleanup → create). `chart.remove()` on cleanup clears lightweight-charts' inline container `width`/`height` styles. The second `createChart()` call reads `clientHeight = 0` and renders nothing. **Mitigation**: always use `getBoundingClientRect()` (forces synchronous layout flush) for initial dimensions, never `clientHeight` alone.

**Do not proceed until all boxes are checked.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| `clientHeight` after `chart.remove()` | `chart.remove()` clears inline styles on the container. A subsequent `createChart()` that reads `clientHeight` gets `0`. This caused the candlestick chart to render invisibly after the `trades` query arrived (firing the effect a second time). | blocking | Use `getBoundingClientRect()` for all initial sizing. `getBoundingClientRect()` forces a synchronous layout flush and returns live dimensions regardless of inline style state. |
| `?? []` in JSX creates new refs | `data ?? []` in JSX creates a new array object every render, causing lightweight-charts `useEffect` deps to differ and the chart to destroy/recreate needlessly. | advisory | Wrap all nullable query results with `useMemo`: `const stableBars = useMemo(() => bars ?? [], [bars])` |
| `autoSize: true` with ResizeObserver | Using `autoSize: true` AND a manual ResizeObserver causes double-sizing. Conflicts with explicit dimension management. | advisory | Pick one: either `autoSize: true` (no ResizeObserver needed) OR explicit sizing + ResizeObserver with `contentRect`. This project uses the explicit path. |
